# Track E — CI/CD & Supply Chain

## Summary
The `claude-skills` repo runs two CI workflows (`validate-frontmatter.yml`, `validate-json-schemas.yml`) that **never declare a `permissions:` block**, so they inherit the repo/org default of `default_workflow_permissions: write` — a serious over-privilege for jobs that only need `contents: read`. Both workflows pin actions by mutable tag (`actions/checkout@v4`, `actions/setup-node@v4`, `actions/upload-artifact@v4`) instead of commit SHA, and `actions/permissions` shows the org does not enforce SHA pinning. Expression interpolation `origin/${{ github.base_ref }}...HEAD` is present in two `run:` blocks; while `github.base_ref` is not directly attacker-controllable on a normal `pull_request` event (it is the *target* branch), the pattern is fragile and would become Critical if the trigger ever changed. **No CodeQL, no `dependency-review-action`, no `npm audit` step, no `SECURITY.md`, no branch protection on `main`, no Dependabot config file** (Dependabot security updates are enabled but version-update config is absent). The `package.json`/`package-lock.json` are out of sync (3.5.0 vs 3.2.3), and `fix-frontmatter.mjs` hardcodes `REPO_ROOT = '/Users/eddie/...'`, making it silently no-op for any other maintainer.

Confirmed via `gh api`: only these two workflow files exist. No `pull_request_target`, `workflow_dispatch`, `repository_dispatch`, `secrets.*` references, or `github.event.{issue,pull_request,comment}.body` interpolations are present in any workflow (grep returned none). Org-level: `default_workflow_permissions: write`, `allowed_actions: all`, `sha_pinning_required: false`. Branch protection on `main`: **none** (404 "Branch not protected"). GitHub secret scanning: **enabled** (2 historical false-positives resolved). Dependabot security updates: **enabled** at the repo level, but no `.github/dependabot.yml` exists (so no version-update PRs for actions or npm).
