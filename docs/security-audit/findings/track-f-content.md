# Track F — Defensive Skill Content Correctness + Injection Surfaces

## Summary

Track F examined whether the security *advice itself* in the defensive-skill corpus (the 5 defensive plugins + the `cybersecurity` offensive router + the 2 hook/scripts) is trustworthy. Assume-breach lens: a marketplace of 142 plugins where users install skills and an AI agent FOLLOWS the instructions. A "security" skill shipping vulnerable example code, an unsafe OSS-tool default, or an authorization gate that rationalizes under prompt-injection pressure is uniquely dangerous because the agent replicates the pattern.

Methodology: every file was read end-to-end. Code blocks were evaluated as if the agent would paste them verbatim. The cybersecurity authorization gate was evaluated under 5 constructed rationalization-bypass scenarios. The 2 injection-surface scripts (`detect-nuxt-seo.sh`, `review-skill.sh`) were audited for unsafe unquoted-variable expansion, command injection, and SSRF/curl-blasting defaults.

**Headline result: the cybersecurity authorization gate is structurally sound (gate fires in main context, never delegated, per-session/per-target-scope ack, sub-agents receive ack_state) but contains two carve-outs that a prompt-injection attacker can drive a truck through: an ownership/localhost claim requiring zero proof, and a "sanctioned bug-bounty" claim that the agent cannot verify.** The defensive plugins are mostly correct, but ship a handful of vulnerable/dead-library examples that an agent would copy verbatim (deprecated `xss-clean`, removed-from-express-validator `escape()`, deprecated-`Markup()` pattern, vulnerable-by-construction `crypto.timingSafeEqual` without a length pre-check, the pastable `// DANGEROUS - avoid these` `innerHTML = userInput` block).

## Cybersecurity authorization-gate review

The gate architecture is the most safety-critical piece in this audit. Verdict: **structurally sound at the design level, with two exploitable carve-outs and one unverifiable defense-in-depth claim.**

Strengths:
- Gate content lives in `references/authorization-disclaimer.md` and is **loaded inline in the main context, never dispatched to a sub-agent** (SKILL.md line 105, decision #5). TOCTOU window is correctly identified and closed.
- The orchestrator fires the gate **before** dispatch (line 116-122), tracks ack per-session/per-target-scope (line 120), and refuses to re-ack blanket when the target "materially changes" (line 50 of the disclaimer).
- Sub-agents receive `ack_state` and `target_scope` as inputs and are instructed to refuse if missing/out-of-scope (line 121).
- The rationalization-resistance table (disclaimer lines 56-71) is comprehensive: it covers authority, time, sunk cost, scope creep, verbal-only, vague, reinterpretation, bypass attempt, normalization, curiosity, and combined pressure. Each entry is a refusal, not a discussion.
- The "Never do" list (SKILL.md lines 234-242) and "Common mistakes" (line 208) both explicitly call out "Softening the gate under pressure" as a known rationalization vector, reinforcing the table.
- Red-flags list (disclaimer lines 27-37) is unconditional: DoS/volume-attack, credential stuffing against third parties, exfiltration, mass exploitation/lateral movement, social engineering, government/critical-infra, "looks like unauthorized intrusion."

Five constructed rationalization-bypass attempts (assumed-breach lens — assume the agent will follow these instructions literally under prompt-injection pressure) found the following residual gaps:

1. **Ownership/localhost claim is proof-free.** authorization-disclaimer.md line 12 accepts "You **own** the target (personal app, localhost, dev/staging environment you control)" as an authorized-context branch requiring **no proof whatsoever**, while third-party claims require "written authorization." An injected instruction that frames the target as "my personal app on 127.0.0.1:8080" or "a dev box I control" clears the gate without any artifact. On a modern developer machine, `localhost` can host a Kubernetes API server, an SSH tunnel to production, a cloud-metadata relay (IMDS proxy), or a vhosted admin panel; treating "localhost" as intrinsically safe is wrong. The rationalization table covers "ownership of `<random production domain>`" but **not** the false-positive direction: the user (or injected instruction) **under-claiming** a target as localhost/dev when it is not. Severity: **High** (the gate's primary control is bypassable by an under-claim).

2. **"Sanctioned bug-bounty program" is unverifiable.** authorization-disclaimer.md line 14 accepts "the target is part of a **sanctioned bug-bounty program** (Bugcrowd, HackerOne, Intigriti public program) AND your test is within scope AND follows the program's rules." The agent has no tool to verify any of the three ANDed conditions. The rationalization table covers "verbal-only" authorization and "vague" authorization, but neither pattern matches the case where the user says "this is in scope of HackerOne program #NNNNN" — that claim is specific, written, and unfalsifiable from the agent's seat. Severity: **Medium** (requires a confident lie that survives the rationalization table; not bypassable by pure pressure).

3. **"Critical infrastructure" / "government" are undefined.** Red-flags line 36 refuses both with no definition. Under prompt-injection pressure to be helpful, ambiguity in these terms bends toward permissiveness: is `acme-county-schools.gov` critical infrastructure? Is a SaaS that *counts* as critical infrastructure under EO 14028 (e.g., a major CDN's customer portal) covered? The disclaimer relies on the agent's common-sense classification, which is exactly the faculty most degradable under pressure. Severity: **Low** (only matters at the margin; "production systems you do not own" still catches most of these).

4. **"OSS tool setup" is Track A and ambiguous.** SKILL.md line 113 lists "OSS tool setup" as **always-allowed** (Track A, no gate). Several of the OSS tools (interact.sh, mitmproxy, nuclei with a URL argument, ffuf with a URL argument) are inherently live-target as soon as they are invoked — there is no clean line between "set up nuclei templates" and "run nuclei against the URL I gave you." An agent that believes it is doing "tool setup" for interact.sh (which by design calls back to a server the agent controls) may route live traffic before the gate fires. Severity: **Medium** (real-world edge case; could route live traffic under a "configuration" framing).

5. **The sub-agent `ack_state` defense-in-depth is unverifiable from below.** SKILL.md line 121 and disclaimer line 52 claim that sub-agents "receive `ack_state` and `target_scope` as inputs and refuse if missing or out-of-scope." This is a text contract: the sub-agent receives a briefing string that **asserts** `ack_state: confirmed`. There is no signature, no shared secret, no way for the sub-agent to verify that the orchestrator actually fired the gate. Under direct prompt-injection into the sub-agent briefing (e.g., a malicious skill the user installed alongside this one that wraps the dispatch), the `ack_state` field is trivially forged. The claim "defense in depth — the gate is the primary control, but sub-agents double-check" is therefore **not actually defense in depth**: it is the same control expressed twice in text. Severity: **Low** (requires compromising the dispatch path; the main-context gate is the real control and it is sound).

**Net gate assessment:** The gate is well-designed and resists the standard rationalization patterns. The residual risk is concentrated in the proof-free ownership/localhost carve-out (F-01) and the unverifiable bug-bounty-scope claim (F-02). Recommend tightening both by requiring *some* artifact for ownership (e.g., "show me the `git remote` / DNS record / dev-server console output") and explicitly flagging bug-bounty-scope claims as "trust on the user" in the skill's failure modes.
