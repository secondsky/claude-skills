# Track C — Secret & Credential Handling

## Summary

`claude-skills` is a marketplace of 142 plugins whose skills routinely demonstrate secret handling (Cloudflare service tokens, JWTs, `BETTER_AUTH_SECRET`, `wrangler secret put`, `.dev.vars`, API keys, HMAC signing keys). Most of the ~90+ `process.env` reads are READ-only and safe; most hardcoded-secret regex hits are placeholders or explicit "Bad" examples (false positives). The genuine problems fall into three defense-in-depth categories: (1) example/sample code that **actively leaks** credentials (logs `console.log(env.API_KEY)`, logs `Verification code: ${token}`, writes API keys into `~/.bashrc` in plaintext); (2) the `better-auth/scripts/generate-secret.sh` generator that **always echoes** the freshly generated secret to stdout with no redaction guidance — an agent-wrapped execution will record that secret in the transcript, log files, or commit it; (3) the `cloudflare-zero-trust-access/scripts/test-access-jwt.sh` "Token is valid" output that performs **only a `kid` presence check, not a cryptographic signature verification**, which contradicts the production template in the same plugin that verifies the signature correctly. No real production secrets were found committed in the repo (only `.env.example` and `.dev.vars.example` files exist), but `.gitignore` does **not** exclude `.dev.vars` and the pre-commit hook does **not** scan for accidentally committed secrets, leaving a meaningful residual risk in an agent-driven workflow.

Notes on what was checked and cleared:

- All ~90+ `process.env.*` reads in `plugins/**/*.{ts,js,mts,mjs,sh}` are READ-only (used for branching/configuration); none are passed to `console.log`, `echo`, `process.stdout.write`, `JSON.stringify`, or returned in HTTP responses.
- All `wrangler secret put` invocations in the repo use the safe stdin-piping pattern (`echo "$X" | wrangler secret put …` or `wrangler secret put … <<< "$X"`). Only C-010 documents the residual hygiene gaps of one such call.
