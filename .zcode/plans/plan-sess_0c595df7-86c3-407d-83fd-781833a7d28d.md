# Plan: Write a phased plan for an OSS-only `cybersecurity` skill

## What this plan produces

A new **top-level** `plans/cybersecurity-skill/` directory in the repo (not `.zcode/plans/` — that's session-scoped). It contains **one orchestration doc** plus **ten phased subplans**. No skill code is written in this pass — only plan documents. The plan documents are detailed enough that a future agent (or you) can execute them phase-by-phase.

6. **Style note** — plan docs match the existing `.zcode/plans/` voice (terse, first-person, engineering-focused), not the README's marketing voice.

### phase-4-authorization-gate.md
Implements the "full offensive + disclaimer" policy:
- Authorization-disclaimer content (authorized testing only — own app / written scope / CTF / sanctioned bug-bounty; legality is user's responsibility; refuse clearly-unauthorized targets like production systems you don't own).
- Two-track routing in SKILL.md:
  - **Always-allowed (no gate)**: static code review, SAST, threat modeling, secure-design review, OSS tool setup, reference lookups, detection-rule authoring.
  - **Gated (requires ack)**: any live-target testing — ffuf/nuclei/ZAP/dalfox/interact.sh against a real host.
- Gate mechanics: a short ack block the user reads; skill proceeds only after ack. No persistent state — gate fires once per session.
- Safety red-flags list (refuse these even with ack): targeting systems you don't own/have no written scope for, DoS-volume requests, credential stuffing against third parties, anything that looks like unauthorized intrusion.

### phase-5-adapt-seven-skills.md
Per-skill porting subplans — one section per named skill. Each section has:
- **Source** (skills.sh URL + raw GitHub URL)
- **What it does** (1-2 lines)
- **Paid → OSS swap table** (specific replacement per paid tool)
- **Porting notes** (what changes in the methodology text, examples, tools table)
- **Pitfalls / gaps** (where OSS parity is imperfect — e.g. XSS DOM Invader has no clean OSS equivalent; document the manual source/sink analysis fallback)

Seven sections:
1. **testing-for-business-logic-vulnerabilities** — Turbo Intruder → `httpracer` / concurrent curl background jobs (skill already shows this pattern); Sequencer → `ent` or drop; Postman → Hoppscotch (OSS). Burp Repeater → mitmproxy/ZAP Request tab.
2. **testing-for-xss-vulnerabilities-with-burpsuite** (hardest) — Burp Scanner → OWASP ZAP active scan + Dalfox; Repeater → ZAP Request tab; Intruder → ffuf/ZAP Fuzzer; DOM Invader → **acknowledge gap**, document manual source/sink analysis + browser DevTools + ZAP DOM XSS add-on (limited); Hackvertor → manual encoding snippets; XSS Hunter → self-hosted BlindXSS fork or skip.
3. **testing-for-host-header-injection** — Burp Collaborator → interact.sh (skill already names this); param-miner → ffuf with header wordlists + custom wordlist; Burp Repeater → mitmproxy.
4. **bypassing-authentication-with-forced-browsing** (easiest) — Burp already optional; drop the Burp Intruder step, replace with ffuf -mc 200 pattern. Rest is already ffuf/Gobuster/SecLists.
5. **testing-for-open-redirect-vulnerabilities** — Burp Collaborator → interact.sh; Burp crawl → OWASP ZAP spider; rest already OpenRedireX/gf/nuclei/ffuf.
6. **cybersecurity-analyst (amplihack)** — pure reasoning framework; **zero tool swaps in the methodology**. Replace the "Security Tools and Platforms" reference inventory at the bottom with OSS equivalents (Nessus → OpenVAS/Vuls; SonarQube → Semgrep; Burp → ZAP; Splunk → Wazuh/Elastic OSS; CrowdStrike → Wazuh; Prisma Cloud → kube-bench/checkov; Recorded Future → MISP/OTX). Keep the 11-step process, STRIDE/PASTA/VAST, MITRE ATT&CK mapping, CVSS/FAIR, NIST IR lifecycle untouched.
7. **security-detections-mcp** — gets its own phase 7.

### phase-7-detections-mcp.md

- **Path A — local OSS MCP** (recommended for OSS-only):

  - MCP client config blocks (Claude Desktop / Cursor / VS Code / ZCode) — all OSS clients, MCP protocol is OSS.
  - Document the 11 named expert prompts (ransomware-readiness, APT emulation, purple-team, etc.) as workflow patterns.
  - Document ATT&CK Navigator layer export.
  - Document the autonomous CTI→detection→PR pipeline (`configure_autonomous` → `run_autonomous_analysis` with cti_sources=[misp,otx]).
- **Path B — hosted API** (optional, documented but not required):
  - `mcp-remote` connection to detect.michaelhaag.org.
  - Account + `sdmcp_` API token from /account/tokens.
  - Free tier documented (200 calls/day, read-only) with explicit note that this is a third-party hosted service.
- **OS adaptation notes**: Sigma-only operation is fully OSS (drop the commercial SIEM dialects if you don't run those SIEMs). The rule *text* in Splunk/CrowdStrike/Sublime/KQL repos is publicly cloned, so reading them costs nothing — running them in the actual SIEM is what's commercial.

### phase-8-defensive-cross-refs.md
Wires up cross-references to the 5 existing defensive plugins (no edits to those plugins — only references FROM the new skill TO them):
- Found XSS → `xss-prevention` skill for remediation.
- Found CSRF → `csrf-protection`.
- Missing security headers → `security-headers-configuration`.
- Need input-validation hardening → `defense-in-depth-validation`.
- Dependency CVE / SBOM / pre-deploy scan → `vulnerability-scanning`.
- Produces a `references/defensive-cross-refs.md` table mapping each finding type → existing skill slug → one-line "what it does for you". Uses `SEE:` markers per writing-skills convention.

### phase-9-testing.md
Applies the `writing-skills` TDD-for-skills discipline to the merged skill:
- **Test approach per skill type** (discipline / technique / pattern / reference) per writing-skills spec.
- **2-3 realistic test prompts per reference doc** (the kind of thing a user would actually type — concrete file paths, casual phrasing).
- **Baseline (RED)** scenarios for the authorization gate: pressure-test that the gate refuses unauthorized live-target testing even under "I'm the owner, trust me" / "just this once" / "I have verbal authorization" pressures.
- **GREEN**: verify each reference doc loads correctly via progressive disclosure and produces correct guidance.
- **REFACTOR**: close loopholes in the authorization gate based on baseline results.
- Test-prompt examples for: XSS testing on own app, business-logic review of an e-commerce flow, threat-modeling a new feature, finding open-redirect in OAuth, Sigma rule for ransomware readiness.
- Verification checklist from writing-skills (frontmatter, description triggers, token budget under 500 lines for SKILL.md, no narrative storytelling).
